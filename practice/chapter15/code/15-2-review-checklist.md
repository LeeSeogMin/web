# 코드 리뷰 체크리스트

## 리뷰어용 체크리스트

### 1. 기능 검증
- [ ] PR 설명에 명시된 대로 동작하는가?
- [ ] 인수 조건(Acceptance Criteria)을 만족하는가?
- [ ] 엣지 케이스가 처리되었는가?
- [ ] 에러 상황이 적절히 처리되는가?

### 2. 코드 품질
- [ ] 코드가 읽기 쉽고 이해하기 쉬운가?
- [ ] 변수/함수 이름이 명확한가?
- [ ] 불필요한 코드(console.log, 주석 처리된 코드)가 없는가?
- [ ] 코드 중복이 없는가?
- [ ] 컴포넌트 분리가 적절한가?

### 3. 스타일 일관성
- [ ] 기존 코드 컨벤션을 따르는가?
- [ ] import 순서가 일관적인가?
- [ ] 들여쓰기, 공백 사용이 일관적인가?
- [ ] 파일/폴더 구조가 프로젝트 규칙을 따르는가?

### 4. 보안 검토
- [ ] 하드코딩된 시크릿(API 키, 비밀번호)이 없는가?
- [ ] 사용자 입력이 검증되는가?
- [ ] XSS 취약점이 없는가? (dangerouslySetInnerHTML 사용 확인)
- [ ] 권한 검사가 적절한가?

### 5. 성능 검토
- [ ] 불필요한 리렌더링이 없는가?
- [ ] 무한 루프 가능성이 없는가?
- [ ] useEffect 의존성 배열이 올바른가?
- [ ] 대용량 데이터 처리 시 페이지네이션이 있는가?

### 6. AI 생성 코드 검증 (해당 시)
- [ ] 사용된 API가 현재 라이브러리 버전과 호환되는가?
- [ ] deprecated 메서드 사용이 없는가?
- [ ] import 경로가 프로젝트 구조와 일치하는가?
- [ ] 공식 문서와 대조 검증했는가?

## 리뷰 코멘트 작성 가이드

### 코멘트 유형 접두사

| 접두사 | 의미 | 예시 |
|--------|------|------|
| `[필수]` | 반드시 수정 필요 | `[필수] XSS 취약점이 있습니다` |
| `[제안]` | 개선 제안, 선택적 | `[제안] 이 로직을 커스텀 훅으로 분리하면 재사용성이 높아질 것 같습니다` |
| `[질문]` | 이해를 위한 질문 | `[질문] 이 조건문의 의도가 무엇인가요?` |
| `[칭찬]` | 좋은 코드 인정 | `[칭찬] 에러 처리가 깔끔합니다!` |

### 좋은 리뷰 코멘트 예시

```markdown
❌ 나쁜 예: "이거 왜 이렇게 했어요?"

✅ 좋은 예: "[제안] 이 부분을 useMemo로 감싸면 불필요한 재계산을
방지할 수 있을 것 같습니다. 현재 방식도 동작하지만, 리스트가
길어지면 성능에 영향을 줄 수 있어요."
```

```markdown
❌ 나쁜 예: "에러 처리 안 됨"

✅ 좋은 예: "[필수] fetchPosts가 실패할 경우 에러 상태를
설정하지 않아 사용자에게 피드백이 없습니다.
catch 블록에서 setError(error.message)를 추가해주세요."
```

## 리뷰 진행 플로우

```
1. PR 설명 읽기
   ↓
2. 변경된 파일 목록 확인
   ↓
3. 전체 diff 훑어보기
   ↓
4. 파일별 상세 리뷰
   ↓
5. 로컬에서 실행 테스트 (필요 시)
   ↓
6. 리뷰 결과 제출
   - Approve: 승인
   - Request Changes: 수정 요청
   - Comment: 의견만 남김
```

## 리뷰어 에티켓

### ✅ DO
- 빠르게 리뷰 시작 (2시간 이내 권장)
- 구체적인 피드백 제공
- 좋은 코드는 칭찬하기
- 코드에 집중, 사람 비판 금지
- 개선 방법 제안 포함

### ❌ DON'T
- "이상함", "별로" 같은 모호한 피드백
- 개인 스타일 강요
- 과도한 리뷰 코멘트 (핵심에 집중)
- 리뷰 없이 방치
- 감정적인 표현 사용
