# 15장 프로젝트 구현과 검증

## 학습목표

1. 기능별 구현 순서를 결정하고 점진적으로 통합할 수 있다
2. PR을 작성하고 코드 리뷰를 수행할 수 있다
3. 기능 테스트와 RLS 검증을 수행할 수 있다
4. README와 회고 보고서를 작성할 수 있다

## 15.1 스프린트 실행

### 15.1.1 기능별 구현 순서

14장에서 설계한 프로젝트를 실제로 구현할 차례다. 어떤 기능부터 구현할지 순서를 정하는 것이 중요하다. 순서를 잘못 정하면 나중에 큰 수정이 필요할 수 있다.

**의존성 기반 구현 순서**:

```
1. 환경 설정 (프로젝트 생성, Supabase 연결)
      ↓
2. 인증 (로그인/로그아웃)
      ↓
3. CRUD 기본 (목록, 상세, 작성)
      ↓
4. CRUD 심화 (수정, 삭제)
      ↓
5. 보안 (RLS 정책)
      ↓
6. UX 개선 (에러 처리, 로딩)
      ↓
7. 배포
```

이 순서를 따르는 이유:

| 단계 | 이유 |
|------|------|
| 환경 설정 먼저 | 모든 기능의 기반 |
| 인증 다음 | 작성자 정보가 필요한 기능들의 전제 조건 |
| 읽기 먼저 | 데이터 없이 쓰기 테스트 어려움 |
| 보안은 기능 후 | 기능이 동작해야 권한 테스트 가능 |
| UX는 마지막 | 핵심 기능 완성 후 다듬기 |

_스프린트 실행 흐름은 content/graphics/ch15/fig-15-1-sprint-flow.mmd 참고_

### 기능 완료 기준

각 기능을 "완료"로 판단하는 기준을 미리 정한다:

```markdown
□ 로컬에서 정상 동작
□ 콘솔 에러 없음
□ 인수 조건(Acceptance Criteria) 충족
□ 코드 리뷰 완료
□ 머지 완료
```

### 15.1.2 점진적 통합

한 번에 모든 기능을 만들고 통합하면 문제가 생겼을 때 원인을 찾기 어렵다. **점진적 통합**(Incremental Integration)은 작은 단위로 자주 통합하는 방식이다.

**작은 단위 커밋 원칙**:

```bash
# ❌ 나쁜 예: 한 번에 많은 변경
git commit -m "게시판 기능 완성"

# ✅ 좋은 예: 작은 단위로 나눠서
git commit -m "feat: PostList 컴포넌트 생성"
git commit -m "feat: 게시글 목록 API 연동"
git commit -m "style: PostList 스타일링"
git commit -m "fix: 빈 목록 처리 추가"
```

**커밋 메시지 컨벤션**:

| 접두사 | 의미 |
|--------|------|
| `feat` | 새 기능 |
| `fix` | 버그 수정 |
| `refactor` | 리팩토링 (기능 변경 없음) |
| `style` | 스타일/CSS 변경 |
| `docs` | 문서 수정 |
| `chore` | 빌드, 설정 등 기타 |

### 빌드 검증

커밋 전 항상 빌드가 성공하는지 확인한다:

```bash
npm run build
```

빌드 실패 상태로 커밋하면 다른 팀원도 영향을 받는다. 빌드 성공 후에만 푸시한다.

## 15.2 코드 리뷰

### 15.2.1 PR 작성법

**Pull Request**(PR)는 코드 변경사항을 메인 브랜치에 병합하기 전에 리뷰를 받는 과정이다. 좋은 PR은 리뷰어가 빠르게 이해하고 피드백할 수 있게 한다.

**PR 작성 흐름**:

1. 기능 브랜치 생성
2. 코드 작성 및 커밋
3. GitHub에 Push
4. PR 생성
5. 리뷰 요청
6. 피드백 반영
7. 승인 후 머지

_PR 워크플로우는 content/graphics/ch15/fig-15-2-pr-workflow.mmd 참고_

**PR 제목 형식**:

```
[타입] 간단한 설명

예시:
[feat] 게시글 작성 기능 구현
[fix] 로그인 후 리다이렉트 오류 수정
[refactor] PostList 컴포넌트 분리
```

**PR 설명 필수 항목**:

| 항목 | 내용 |
|------|------|
| **변경 사항** | 무엇이 변경되었는지 |
| **관련 이슈** | 연결된 이슈 번호 |
| **스크린샷** | UI 변경 시 전후 비교 |
| **테스트 방법** | 리뷰어가 테스트할 방법 |
| **체크리스트** | 자체 검증 항목 |

**좋은 PR의 조건**:

- **작고 집중적**: 한 PR에 한 기능 (200줄 이내 권장)
- **명확한 설명**: 왜 이 변경이 필요한지
- **시각적 데모**: 스크린샷, GIF 첨부
- **자체 검증 완료**: 제출 전 직접 테스트

_PR 템플릿은 practice/chapter15/code/15-2-pr-template.md 참고_

### 15.2.2 리뷰 체크리스트

코드 리뷰 시 확인해야 할 항목을 체크리스트로 정리한다.

**기능 검증**:

- [ ] PR 설명대로 동작하는가?
- [ ] 인수 조건을 만족하는가?
- [ ] 엣지 케이스가 처리되었는가?

**코드 품질**:

- [ ] 코드가 읽기 쉬운가?
- [ ] 변수/함수 이름이 명확한가?
- [ ] 불필요한 코드(console.log 등)가 없는가?
- [ ] 코드 중복이 없는가?

**보안 검토**:

- [ ] 하드코딩된 시크릿이 없는가?
- [ ] 사용자 입력이 검증되는가?
- [ ] XSS 취약점이 없는가?

**AI 생성 코드 검증**:

- [ ] 사용된 API가 현재 라이브러리 버전과 호환되는가?
- [ ] deprecated 메서드 사용이 없는가?
- [ ] import 경로가 프로젝트 구조와 일치하는가?

### 리뷰 코멘트 작성 가이드

리뷰 코멘트에 접두사를 붙이면 의도가 명확해진다:

| 접두사 | 의미 |
|--------|------|
| `[필수]` | 반드시 수정 필요 |
| `[제안]` | 개선 제안, 선택적 |
| `[질문]` | 이해를 위한 질문 |
| `[칭찬]` | 좋은 코드 인정 |

**좋은 코멘트 예시**:

```markdown
❌ 나쁜 예: "이상함"

✅ 좋은 예: "[제안] 이 로직을 useMemo로 감싸면
   불필요한 재계산을 방지할 수 있을 것 같습니다."
```

_리뷰 체크리스트는 practice/chapter15/code/15-2-review-checklist.md 참고_

## 15.3 최종 검증

### 15.3.1 기능 테스트

배포 전 모든 기능이 정상 동작하는지 확인한다. 테스트 시나리오를 미리 작성하고 하나씩 확인한다.

**인증 기능 테스트**:

| 시나리오 | 기대 결과 | 통과 |
|----------|-----------|------|
| 로그인 버튼 클릭 | OAuth 페이지로 이동 | ☐ |
| 로그인 성공 | 홈으로 리다이렉트, 사용자 정보 표시 | ☐ |
| 로그아웃 | 세션 종료, 로그인 버튼 표시 | ☐ |
| 비로그인 상태로 보호된 페이지 접근 | 로그인 페이지로 이동 | ☐ |

**CRUD 기능 테스트**:

| 시나리오 | 기대 결과 | 통과 |
|----------|-----------|------|
| 목록 조회 | 게시글 목록 표시, 최신순 정렬 | ☐ |
| 상세 조회 | 제목, 내용, 작성자 표시 | ☐ |
| 게시글 작성 | 저장 후 목록에 표시 | ☐ |
| 게시글 수정 | 변경 내용 반영 | ☐ |
| 게시글 삭제 | 목록에서 제거 | ☐ |

_테스트 유형 다이어그램은 content/graphics/ch15/fig-15-3-test-types.mmd 참고_

### 15.3.2 RLS 검증

11장에서 작성한 RLS 정책이 실제로 동작하는지 검증한다. 이 단계가 매우 중요하다. RLS가 제대로 동작하지 않으면 다른 사용자의 데이터가 노출될 수 있다.

**Anonymous 테스트** (비로그인 상태):

| 시나리오 | 기대 결과 | 통과 |
|----------|-----------|------|
| 목록 조회 | ✅ 조회 가능 | ☐ |
| 게시글 작성 시도 | ❌ 차단, 로그인 유도 | ☐ |
| 게시글 수정 시도 | ❌ 차단 | ☐ |

**Member 테스트** (일반 회원):

| 시나리오 | 기대 결과 | 통과 |
|----------|-----------|------|
| 자신의 게시글 수정 | ✅ 가능 | ☐ |
| 자신의 게시글 삭제 | ✅ 가능 | ☐ |
| 다른 사람 게시글 수정 | ❌ 차단 | ☐ |
| 다른 사람 게시글 삭제 | ❌ 차단 | ☐ |

**검증 방법**:

1. 테스트용 계정 2개 준비 (사용자 A, 사용자 B)
2. 사용자 A로 게시글 작성
3. 사용자 B로 로그인하여 A의 게시글 수정/삭제 시도
4. 권한 오류 발생 확인

### 15.3.3 에러 시나리오 테스트

정상 흐름뿐 아니라 에러 상황도 테스트한다.

**네트워크 오류**:

| 시나리오 | 기대 결과 | 통과 |
|----------|-----------|------|
| 오프라인에서 조회 | 에러 메시지, 재시도 버튼 | ☐ |
| API 응답 지연 | 로딩 인디케이터 | ☐ |
| 서버 500 오류 | 사용자 친화적 메시지 | ☐ |

**입력 오류**:

| 시나리오 | 기대 결과 | 통과 |
|----------|-----------|------|
| 빈 제목으로 작성 | 유효성 오류 메시지 | ☐ |
| 제목 200자 초과 | 길이 제한 안내 | ☐ |
| XSS 스크립트 입력 | 무해화 처리 | ☐ |

**세션 오류**:

| 시나리오 | 기대 결과 | 통과 |
|----------|-----------|------|
| 세션 만료 후 작성 | 로그인 페이지로 이동 | ☐ |

_전체 테스트 시나리오는 practice/chapter15/code/15-3-test-scenarios.md 참고_

## 15.4 문서화

### 15.4.1 README 작성

**README**는 프로젝트의 "첫인상"이다. 다른 개발자가 프로젝트를 처음 볼 때 가장 먼저 읽는 문서다.

**README 필수 섹션**:

| 섹션 | 내용 |
|------|------|
| **프로젝트 소개** | 무엇을 하는 프로젝트인지 |
| **주요 기능** | 핵심 기능 목록 |
| **기술 스택** | 사용한 기술 |
| **시작하기** | 설치 및 실행 방법 |
| **환경 변수** | 필요한 환경 변수 설명 |
| **프로젝트 구조** | 폴더 구조 설명 |

**시작하기 섹션 예시**:

```markdown
## 🚀 시작하기

### 사전 요구사항
- Node.js 18.0.0 이상
- Supabase 계정

### 설치
```bash
git clone https://github.com/username/project.git
cd project
npm install
cp .env.example .env.local
npm run dev
```
```

**환경 변수 문서화**:

```markdown
## 🔐 환경 변수

| 변수명 | 설명 | 필수 |
|--------|------|------|
| `VITE_SUPABASE_URL` | Supabase 프로젝트 URL | ✅ |
| `VITE_SUPABASE_ANON_KEY` | Supabase Anonymous Key | ✅ |

⚠️ `.env.local` 파일은 절대 Git에 커밋하지 마세요!
```

_README 템플릿은 practice/chapter15/code/15-4-readme-template.md 참고_

### 15.4.2 AI 사용 로그 정리

프로젝트에서 AI를 어떻게 활용했는지 기록한다. 이는 학습 과정을 정리하고, 향후 프로젝트에서 더 효과적으로 AI를 활용하는 데 도움이 된다.

**AI 사용 로그 항목**:

| 항목 | 내용 |
|------|------|
| **사용한 프롬프트** | AI에게 요청한 내용 |
| **AI 출력** | 생성된 코드/답변 |
| **검증 결과** | 정상 동작 여부 |
| **수정 내역** | 직접 수정한 부분 |

**예시**:

```markdown
## AI 사용 로그

### 1. PostList 컴포넌트 생성

**프롬프트**:
"React 18, Supabase JS v2 환경에서 게시글 목록을
조회하는 PostList 컴포넌트를 작성해줘"

**AI 출력**: (코드 생략)

**검증 결과**: ✅ 정상 동작

**수정 내역**:
- import 경로 수정: `supabase` → `@/lib/supabase`
- 로딩 상태 추가
```

### AI 활용 통계

프로젝트 완료 후 AI 활용을 정량화한다:

| 항목 | 수치 |
|------|------|
| AI로 생성한 코드 비율 | 약 40% |
| 수정이 필요했던 비율 | 약 30% |
| 가장 유용했던 활용 | 보일러플레이트 생성 |

### 15.4.3 회고 보고서

**회고**(Retrospective)는 프로젝트를 돌아보며 배운 점과 개선할 점을 정리하는 활동이다.

**Start, Stop, Continue 형식**:

| 분류 | 내용 |
|------|------|
| **Start** | 다음에 새로 시작할 것 |
| **Stop** | 다음에 하지 않을 것 |
| **Continue** | 잘 되어서 계속할 것 |

**예시**:

```markdown
## Start (새로 시작할 것)
- PR 템플릿 사용하기
- 커밋 전 린트 검사 자동화

## Stop (멈출 것)
- 큰 기능을 한 번에 커밋하기
- 테스트 없이 머지하기

## Continue (계속할 것)
- 작은 단위로 커밋하기
- AI 출력 검증 체크리스트 사용
```

**학습 내용 정리**:

| 주제 | 배운 내용 |
|------|-----------|
| React | useEffect 의존성 배열 관리 |
| Supabase | RLS 정책 작성 및 테스트 |
| 배포 | Vercel 환경 변수 관리 |
| AI | 버전 명시 프롬프팅의 중요성 |

**다음 프로젝트 적용 계획**:

```markdown
1. 프로젝트 시작 전 AI 협업 계획 수립
2. copilot-instructions.md 미리 작성
3. RLS 정책은 기능 구현과 동시에 테스트
4. PR 체크리스트 적극 활용
```

_회고 보고서 템플릿은 practice/chapter15/code/15-4-retrospective-template.md 참고_

## 장 요약

- **구현 순서**는 의존성을 고려하여 인증 → CRUD → 보안 순으로 진행한다
- **점진적 통합**으로 작은 단위의 커밋을 자주 수행한다
- **PR**은 작고 집중적으로 작성하고, 스크린샷과 테스트 방법을 포함한다
- **코드 리뷰**는 기능, 품질, 보안, AI 코드 검증을 체크리스트로 확인한다
- **RLS 검증**은 다른 사용자 계정으로 권한 차단을 직접 테스트한다
- **README**는 설치 방법, 환경 변수, 프로젝트 구조를 명확히 문서화한다
- **회고**로 Start/Stop/Continue를 정리하고 다음 프로젝트에 반영한다

## 연습문제

### 기초

1. 점진적 통합의 장점을 2가지 설명하시오.

2. PR 작성 시 포함해야 할 필수 항목 5가지를 나열하시오.

3. 코드 리뷰 코멘트의 `[필수]`와 `[제안]` 접두사의 차이를 설명하시오.

### 중급

4. 게시판 프로젝트에서 인증 기능을 CRUD보다 먼저 구현해야 하는 이유를 설명하시오.

5. RLS 검증을 위해 "다른 사용자의 게시글 수정 차단"을 테스트하는 구체적인 단계를 작성하시오.

6. README에 환경 변수를 문서화할 때 주의해야 할 점을 설명하시오.

### 심화

7. 다음 시나리오에 대한 테스트 케이스를 작성하시오:
   - 기능: 게시글 삭제
   - 역할: Anonymous, Member(본인), Member(타인), Admin
   - 각 역할별 기대 결과 포함

8. 프로젝트 회고 보고서를 작성하시오. 가상의 게시판 프로젝트를 가정하고, Start/Stop/Continue와 학습 내용, AI 활용 통계를 포함하시오.
