# 부록 B. AI 사용 로그 템플릿

이 부록에서는 AI 도구 활용 내역을 기록하는 방법을 안내한다. AI 사용 로그는 학습 과정을 정리하고, AI 출력의 검증 여부를 추적하는 데 유용하다.

---

## B.1 로그 작성 양식

### 기본 양식

```markdown
# AI 사용 로그

## 프로젝트 정보
- **프로젝트명**:
- **작성자**:
- **기간**: YYYY-MM-DD ~ YYYY-MM-DD

---

## 로그 항목

### [번호] 작업 제목
- **날짜**: YYYY-MM-DD
- **AI 도구**: (Claude / Copilot / ChatGPT 등)
- **작업 유형**: (코드 생성 / 디버깅 / 설명 요청 / 리팩토링)

**프롬프트**:
```
(AI에게 요청한 내용)
```

**AI 출력 요약**:
(생성된 코드나 답변의 핵심 내용)

**검증 결과**: ✅ 정상 / ⚠️ 수정 필요 / ❌ 사용 불가

**수정 내역** (해당 시):
- (수정한 내용)

**교훈**:
- (배운 점이나 주의할 점)

---
```

### 간소화 양식 (표 형식)

| 날짜 | 작업 | AI 도구 | 프롬프트 요약 | 검증 | 비고 |
|------|------|---------|--------------|------|------|
| | | | | ✅/⚠️/❌ | |

## B.2 작성 예시

### 예시 1: 컴포넌트 생성

```markdown
### [001] PostList 컴포넌트 생성
- **날짜**: 2024-03-15
- **AI 도구**: GitHub Copilot
- **작업 유형**: 코드 생성

**프롬프트**:
```
// React 18, Supabase v2
// 게시글 목록을 조회하여 표시하는 컴포넌트
// 로딩 상태와 에러 처리 포함
```

**AI 출력 요약**:
- useState로 posts, loading, error 상태 관리
- useEffect에서 supabase.from('posts').select() 호출
- 조건부 렌더링으로 로딩/에러/목록 표시

**검증 결과**: ⚠️ 수정 필요

**수정 내역**:
- import 경로 수정: `'supabase'` → `'@/lib/supabase'`
- 정렬 옵션 추가: `.order('created_at', { ascending: false })`

**교훈**:
- Copilot은 프로젝트 구조를 모르므로 import 경로 확인 필수
- 기본 쿼리에 정렬이 없으면 순서가 불규칙할 수 있음
```

### 예시 2: 에러 디버깅

```markdown
### [002] RLS 정책 오류 해결
- **날짜**: 2024-03-16
- **AI 도구**: Claude
- **작업 유형**: 디버깅

**프롬프트**:
```
Supabase에서 게시글 작성 시 다음 에러가 발생합니다:
"new row violates row-level security policy"

현재 RLS 정책:
CREATE POLICY "posts_insert" ON posts
FOR INSERT WITH CHECK (auth.uid() = author_id);

문제가 무엇인가요?
```

**AI 출력 요약**:
- INSERT 시 author_id를 명시적으로 설정해야 함
- 클라이언트에서 author_id: user.id를 포함해야 정책 통과

**검증 결과**: ✅ 정상

**교훈**:
- RLS WITH CHECK는 새로 삽입되는 행을 검증함
- 클라이언트에서 author_id를 반드시 포함해야 함
```

## B.3 자주 하는 실수

### 1. 검증 없이 AI 출력 사용

```markdown
❌ 나쁜 예:
AI가 생성한 코드를 바로 복사-붙여넣기

✅ 좋은 예:
1. 생성된 코드의 import 경로 확인
2. 사용된 API가 현재 버전과 호환되는지 확인
3. 실제 실행하여 동작 확인
4. 로그에 검증 결과 기록
```

### 2. 로그 미작성

```markdown
❌ 나쁜 예:
"나중에 정리해야지" → 결국 기록 없음

✅ 좋은 예:
- AI 사용 직후 간단히라도 기록
- 표 형식으로 빠르게 작성
- 중요한 교훈은 상세히 기록
```

### 3. 프롬프트 기록 누락

```markdown
❌ 나쁜 예:
"컴포넌트 만들어달라고 했음" (구체적 내용 없음)

✅ 좋은 예:
프롬프트 전문을 그대로 기록
- 어떤 조건을 명시했는지
- 어떤 버전을 지정했는지
- 어떤 제약을 걸었는지
```

### 4. 수정 내역 미기록

```markdown
❌ 나쁜 예:
"수정함" (무엇을 수정했는지 불명확)

✅ 좋은 예:
- 수정 전 코드와 수정 후 코드 비교
- 왜 수정이 필요했는지 이유 기록
- 다음에 같은 실수를 피할 방법
```

## 로그 활용 팁

### 1. 주기적 검토

주 1회 로그를 검토하며:
- 반복되는 수정 패턴이 있는가?
- 특정 유형의 작업에서 AI가 잘 동작하는가?
- 프롬프트를 개선할 수 있는 부분이 있는가?

### 2. 팀 공유

팀 프로젝트에서는:
- 공통 저장소에 로그 공유
- 다른 팀원의 교훈 참고
- 효과적인 프롬프트 패턴 공유

### 3. 템플릿 커스터마이징

프로젝트 특성에 맞게:
- 필요한 항목 추가 (예: 사용 시간, 토큰 수)
- 불필요한 항목 제거
- 팀 컨벤션에 맞게 수정
