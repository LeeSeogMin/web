## 12.1 에러의 종류

웹 애플리케이션에서 발생하는 에러는 크게 네 가지로 분류할 수 있다.

```
┌─────────────────────────────────────────────────────────────┐
│                    웹 애플리케이션 에러                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌───────────────┐    ┌───────────────┐                   │
│   │  네트워크 에러 │    │   인증 에러    │                   │
│   │ - 연결 끊김   │    │ - 토큰 만료   │                   │
│   │ - 타임아웃    │    │ - 미인증 401  │                   │
│   │ - 서버 5xx   │    └───────────────┘                   │
│   └───────────────┘                                        │
│                                                             │
│   ┌───────────────┐    ┌───────────────┐                   │
│   │   권한 에러    │    │ 유효성 검증   │                   │
│   │ - RLS 위반    │    │ - 필수 필드   │                   │
│   │ - 권한 부족   │    │ - 형식 오류   │                   │
│   │ - 403 에러   │    │ - 중복 데이터  │                   │
│   └───────────────┘    └───────────────┘                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘

※ 그림 12.1 웹 애플리케이션 에러 유형
```

### 12.1.1 네트워크 에러

인터넷 연결이 끊기거나 서버가 응답하지 않을 때 발생한다.

**발생 상황**:
- 와이파이/모바일 데이터 끊김
- 서버 다운 또는 과부하
- 요청 타임아웃

**Supabase에서 발생하는 에러**:
```javascript
// TypeError: Failed to fetch
// NetworkError when attempting to fetch resource
```

**처리 방법**: 재시도 버튼 제공, 오프라인 상태 감지

### 12.1.2 인증 에러

사용자의 로그인 상태에 문제가 있을 때 발생한다.

**발생 상황**:
- 토큰 만료 (Supabase는 기본 1시간)
- 세션 종료
- 로그인 없이 보호된 리소스 접근

**Supabase에서 발생하는 에러**:
```javascript
// AuthSessionMissingError
// HTTP 401 Unauthorized
```

**처리 방법**: 로그인 페이지로 리다이렉트, 토큰 자동 갱신

### 12.1.3 권한 에러

RLS(Row Level Security) 정책에 의해 접근이 거부될 때 발생한다.

**발생 상황**:
- 타인의 게시글 수정/삭제 시도
- 권한 없는 데이터 조회

**Supabase에서 발생하는 에러**:
```javascript
// code: 'PGRST301'
// HTTP 403 Forbidden
```

**주의**: SELECT 정책 위반은 에러가 아니라 빈 결과를 반환한다.

### 12.1.4 유효성 검증 에러

입력 데이터가 올바르지 않을 때 발생한다.

**발생 상황**:
- 필수 필드 누락
- 이메일 형식 오류
- 중복된 데이터 (unique 제약조건)

**Supabase에서 발생하는 에러**:
```javascript
// code: '23505' - unique 위반
// code: '23502' - not null 위반
// code: '23503' - foreign key 위반
```

**표 12.1** 에러 유형별 특징

| 에러 유형 | HTTP 코드 | Supabase 코드 | 대응 방법 |
|----------|----------|---------------|----------|
| 네트워크 | - | NetworkError | 재시도 버튼 |
| 인증 | 401 | Auth* | 로그인 페이지 |
| 권한 | 403 | PGRST301 | 권한 안내 |
| 유효성 | 400 | 23xxx | 입력 필드 표시 |

## 12.2 에러 처리 패턴

### 12.2.1 try-catch 구조화

Supabase 호출은 에러를 throw하지 않고 `{ data, error }` 형태로 반환한다. 이를 throw 패턴으로 변환하면 일관된 에러 처리가 가능하다.

**기본 패턴**:
```javascript
async function fetchPosts() {
    try {
        const { data, error } = await supabase
            .from('posts')
            .select('*');

        if (error) throw error;  // 에러를 throw로 변환

        return data;
    } catch (err) {
        console.error('게시글 조회 실패:', err.message);
        throw err;  // 상위로 전파
    }
}
```

**에러 유형별 분기 처리**:
```javascript
async function handleDelete(postId) {
    try {
        const { error } = await supabase
            .from('posts')
            .delete()
            .eq('id', postId);

        if (error) throw error;
    } catch (err) {
        // 에러 유형별 처리
        if (err.code === 'PGRST301') {
            alert('이 게시글을 삭제할 권한이 없습니다.');
        } else if (err.message.includes('fetch')) {
            alert('네트워크 연결을 확인해주세요.');
        } else {
            alert('삭제에 실패했습니다.');
        }
    }
}
```

### 12.2.2 React Error Boundary

**Error Boundary**는 하위 컴포넌트에서 발생한 JavaScript 에러를 잡아 **대체 UI**를 보여주는 React 컴포넌트다.

```
┌─────────────────────────────────────────────────────────────┐
│                  Error Boundary 동작 원리                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────────────────────────────────┐                   │
│   │          Error Boundary             │                   │
│   │                                     │                   │
│   │   ┌───────────────────────────┐    │                   │
│   │   │     자식 컴포넌트          │    │                   │
│   │   │                           │    │                   │
│   │   │   에러 발생! ──────────────┼────┼──▶ 에러 포착     │
│   │   │                           │    │                   │
│   │   └───────────────────────────┘    │                   │
│   │                                     │                   │
│   └─────────────────────────────────────┘                   │
│                      │                                      │
│                      ▼                                      │
│   ┌─────────────────────────────────────┐                   │
│   │         Fallback UI                 │                   │
│   │   "문제가 발생했습니다."              │                   │
│   │   [다시 시도] [홈으로]               │                   │
│   └─────────────────────────────────────┘                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘

※ 그림 12.2 Error Boundary 동작 원리
```

**기본 구현** (클래스 컴포넌트):
```jsx
class ErrorBoundary extends Component {
    constructor(props) {
        super(props);
        this.state = { hasError: false, error: null };
    }

    static getDerivedStateFromError(error) {
        return { hasError: true, error };
    }

    componentDidCatch(error, errorInfo) {
        console.error('Error caught:', error, errorInfo);
    }

    render() {
        if (this.state.hasError) {
            return (
                <div>
                    <h2>문제가 발생했습니다</h2>
                    <button onClick={() => this.setState({ hasError: false })}>
                        다시 시도
                    </button>
                </div>
            );
        }
        return this.props.children;
    }
}
```

**사용법**:
```jsx
// 전체 앱 감싸기
<ErrorBoundary>
    <App />
</ErrorBoundary>

// 특정 영역만 감싸기
<ErrorBoundary>
    <PostContent />
</ErrorBoundary>
```

**제한사항**:
- 비동기 에러는 잡지 못함 (try-catch 사용)
- 이벤트 핸들러 에러는 잡지 못함
- 클래스 컴포넌트로만 구현 가능

> **팁**: react-error-boundary 라이브러리를 사용하면 함수형 컴포넌트에서도 쉽게 사용할 수 있다.

### 12.2.3 사용자 친화적 에러 메시지

개발자용 에러 메시지를 그대로 보여주면 사용자가 이해할 수 없다.

**나쁜 예**:
```
PGRST301: new row violates row-level security policy for table "posts"
```

**좋은 예**:
```
이 게시글을 수정할 권한이 없습니다.
```

**에러 메시지 매핑**:
```javascript
const ERROR_MESSAGES = {
    'PGRST301': '이 작업을 수행할 권한이 없습니다.',
    'AuthSessionMissingError': '로그인이 필요합니다.',
    '23505': '이미 존재하는 데이터입니다.',
    '23502': '필수 항목이 누락되었습니다.',
    'default': '문제가 발생했습니다. 잠시 후 다시 시도해주세요.'
};

function getUserFriendlyMessage(error) {
    const code = error?.code || '';
    return ERROR_MESSAGES[code] || ERROR_MESSAGES.default;
}
```

**표 12.2** 에러 메시지 변환 예시

| 원본 에러 | 사용자 메시지 |
|----------|-------------|
| PGRST301 | 권한이 없습니다 |
| 23505 | 이미 존재하는 데이터입니다 |
| NetworkError | 인터넷 연결을 확인해주세요 |
| 401 | 로그인이 필요합니다 |

### 12.3.3 Optimistic UI

**Optimistic UI**는 서버 응답 전에 성공을 가정하고 UI를 먼저 업데이트하는 패턴이다.

**장점**:
- 즉각적인 피드백 (대기 시간 0ms)
- 체감 속도 대폭 향상

**구현 패턴**:
```javascript
async function handleDelete(postId) {
    // 1. 현재 상태 저장 (롤백용)
    const previousPosts = [...posts];

    // 2. 낙관적 업데이트 - 즉시 UI 반영
    setPosts(posts.filter(p => p.id !== postId));

    try {
        // 3. 서버에 요청
        const { error } = await supabase
            .from('posts')
            .delete()
            .eq('id', postId);

        if (error) throw error;
    } catch (err) {
        // 4. 실패 시 롤백
        setPosts(previousPosts);
        alert('삭제에 실패했습니다.');
    }
}
```

**표 12.3** 로딩 UI 패턴 비교

| 패턴 | 장점 | 단점 | 사용 시점 |
|------|------|------|----------|
| 스피너 | 구현 간단 | 진행률 불명확 | 짧은 작업 |
| 스켈레톤 | 레이아웃 유지 | 구현 복잡 | 콘텐츠 로딩 |
| Optimistic | 즉각 반응 | 롤백 필요 | 성공률 높은 작업 |

## 12.4 실습: 에러/로딩 UX 개선

### 요구사항

10장에서 만든 게시판에 다음 UX 개선사항을 적용한다:

1. 게시글 목록 로딩 시 스켈레톤 UI
2. 에러 발생 시 사용자 친화적 메시지
3. 삭제 시 Optimistic UI
4. 토스트 알림으로 결과 피드백

### 핵심 코드

**useLoading 커스텀 훅**:
```jsx
function useLoading(initialState = false) {
    const [loading, setLoading] = useState(initialState);
    const [error, setError] = useState(null);

    const execute = async (asyncFn) => {
        try {
            setLoading(true);
            setError(null);
            return await asyncFn();
        } catch (err) {
            setError(err);
            throw err;
        } finally {
            setLoading(false);
        }
    };

    return { loading, error, execute };
}
```

**개선된 게시글 목록**:
```jsx
function EnhancedPostList() {
    const [posts, setPosts] = useState([]);
    const [toast, setToast] = useState(null);
    const { loading, error, execute } = useLoading(true);

    useEffect(() => {
        fetchPosts();
    }, []);

    async function fetchPosts() {
        try {
            await execute(async () => {
                const { data, error } = await supabase
                    .from('posts')
                    .select('*, profiles(username)')
                    .order('created_at', { ascending: false });

                if (error) throw error;
                setPosts(data);
            });
        } catch (err) {
            setToast({
                message: getUserFriendlyMessage(err),
                type: 'error'
            });
        }
    }

    // 로딩 중 - 스켈레톤
    if (loading) return <PostListSkeleton count={5} />;

    // 에러 - 재시도 버튼
    if (error) {
        return (
            <div className="error-state">
                <p>게시글을 불러오는데 실패했습니다.</p>
                <button onClick={fetchPosts}>다시 시도</button>
            </div>
        );
    }

    return (
        <div className="post-list">
            {posts.map(post => (
                <PostCard key={post.id} post={post} />
            ))}
            {toast && <Toast {...toast} onClose={() => setToast(null)} />}
        </div>
    );
}
```

**토스트 컴포넌트**:
```jsx
function Toast({ message, type, onClose }) {
    useEffect(() => {
        const timer = setTimeout(onClose, 3000);
        return () => clearTimeout(timer);
    }, [onClose]);

    return (
        <div className={`toast toast-${type}`}>
            {message}
        </div>
    );
}
```

## 핵심 정리

1. 웹 애플리케이션 에러는 **네트워크, 인증, 권한, 유효성 검증** 네 가지로 분류할 수 있다.

2. **try-catch**로 Supabase 에러를 잡고, 에러 유형에 따라 다르게 처리한다.

3. **Error Boundary**는 컴포넌트 에러를 격리하여 전체 앱이 중단되지 않게 한다. 단, 비동기 에러와 이벤트 핸들러 에러는 잡지 못한다.

4. **사용자 친화적 에러 메시지**로 변환하여 보여준다. 기술적 에러 코드를 그대로 노출하지 않는다.

5. **스켈레톤 UI**는 로딩 중 레이아웃을 미리 보여주어 체감 속도를 높인다.

6. **Optimistic UI**는 서버 응답 전에 UI를 먼저 업데이트한다. 실패 시 롤백이 필수다.

7. 에러/성공 알림은 **토스트**로 잠시 보여주고 자동으로 사라지게 한다.

## 연습문제

### 기초
