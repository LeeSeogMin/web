# 13장 Vercel 배포

## 13.1 Vercel 플랫폼 이해

### 13.1.1 서버리스 호스팅이란

전통적인 웹 호스팅은 서버를 직접 관리해야 했다. 서버 설정, 보안 패치, 트래픽 증가에 따른 스케일링까지 모두 개발자의 몫이었다. **서버리스**(Serverless) 호스팅은 이런 서버 관리 부담을 완전히 제거한다.

쉽게 말해서, 서버리스는 "서버가 없다"는 의미가 아니라 "서버를 신경 쓸 필요가 없다"는 의미다. 클라우드 제공자가 인프라를 자동으로 관리하고, 개발자는 코드에만 집중할 수 있다.

```
전통적 호스팅: 개발자 → 서버 설정 → 배포 → 모니터링 → 스케일링
서버리스:      개발자 → 코드 푸시 → 자동 배포 완료
```

### 13.1.2 Vercel의 주요 특징

**Vercel**은 프론트엔드 개발에 최적화된 서버리스 플랫폼이다. Next.js를 만든 회사답게 React 기반 프로젝트와의 호환성이 뛰어나다.

핵심 특징을 살펴보자:

| 특징 | 설명 |
|------|------|
| **제로 설정 배포** | 프레임워크 자동 감지, 빌드 명령어 추론 |
| **글로벌 CDN** | 전 세계 엣지 네트워크에서 콘텐츠 제공 |
| **프리뷰 배포** | PR마다 별도 URL로 미리보기 환경 생성 |
| **즉시 롤백** | 클릭 한 번으로 이전 버전 복구 |
| **무료 SSL** | HTTPS 인증서 자동 발급 및 갱신 |

Vite로 빌드한 React 프로젝트는 Vercel이 자동으로 인식한다. 별도 설정 없이 `npm run build` 결과물을 CDN에 배포해준다.

### 13.1.3 배포 플랫폼 비교

프론트엔드 배포 플랫폼은 여러 가지가 있다:

| 플랫폼 | 장점 | 단점 | 적합한 용도 |
|--------|------|------|-------------|
| **Vercel** | 최고의 DX, 프리뷰 배포 | 서버 기능 제한적 | React/Next.js 프로젝트 |
| **Netlify** | 폼 처리, 함수 지원 | Vercel 대비 느린 빌드 | JAMstack 사이트 |
| **GitHub Pages** | 완전 무료 | 정적 사이트만 | 문서, 포트폴리오 |
| **AWS Amplify** | AWS 통합 | 설정 복잡 | 엔터프라이즈 앱 |

이 교재에서는 Supabase와의 연동이 간편하고 학습 곡선이 낮은 Vercel을 사용한다.

## 13.2 배포 프로세스

### 13.2.1 배포 흐름 이해

Vercel 배포는 Git 기반 워크플로우를 따른다. 코드를 GitHub에 푸시하면 자동으로 빌드와 배포가 시작된다.

```
[개발자] ──git push──▶ [GitHub] ──웹훅──▶ [Vercel]
                                              │
                                    ┌─────────┴─────────┐
                                    ▼                   ▼
                              의존성 설치          vite build
                                    │                   │
                                    └─────────┬─────────┘
                                              ▼
                                      정적 파일 생성
                                              │
                                              ▼
                                    글로벌 CDN 배포
                                              │
                                              ▼
                                      배포 완료 알림
```

_전체 다이어그램은 content/graphics/ch13/fig-13-1-deploy-flow.mmd 참고_

이 과정을 **CI/CD**(Continuous Integration/Continuous Deployment) 파이프라인이라고 한다. 개발자가 코드를 푸시할 때마다 자동으로 테스트, 빌드, 배포가 이루어진다.

### 13.2.2 GitHub 연동

Vercel 배포의 첫 단계는 GitHub 계정 연동이다.

**1단계: Vercel 가입**

1. [vercel.com](https://vercel.com)에 접속
2. "Start Deploying" 클릭
3. "Continue with GitHub" 선택
4. GitHub 계정으로 로그인 및 권한 승인

**2단계: 저장소 접근 권한 설정**

GitHub 연동 시 두 가지 옵션이 있다:

- **All repositories**: 모든 저장소에 Vercel 접근 허용
- **Only select repositories**: 특정 저장소만 선택

보안을 위해 "Only select repositories"를 권장한다. 배포할 프로젝트만 선택하면 된다.

### 13.2.3 프로젝트 임포트

GitHub 연동이 완료되면 프로젝트를 Vercel로 가져온다.

**임포트 단계**:

1. Vercel 대시보드에서 "Add New..." → "Project" 클릭
2. 연동된 GitHub 저장소 목록에서 프로젝트 선택
3. "Import" 클릭

Vercel이 `package.json`을 분석하여 프레임워크를 자동 감지한다. Vite 프로젝트는 다음과 같이 설정된다:

| 설정 항목 | 자동 감지 값 |
|-----------|--------------|
| Framework Preset | Vite |
| Build Command | `vite build` |
| Output Directory | `dist` |
| Install Command | `npm install` |

대부분의 경우 자동 감지된 설정을 그대로 사용하면 된다.

### 13.2.4 SPA 라우팅 설정

React Router를 사용하는 SPA(Single Page Application)는 추가 설정이 필요하다. 사용자가 `/posts/123` 같은 URL로 직접 접속하면 Vercel은 해당 경로의 파일을 찾으려 한다. 하지만 SPA에서는 모든 경로가 `index.html`로 처리되어야 한다.

`vercel.json` 파일로 이 문제를 해결한다:

```json
{
  "rewrites": [
    { "source": "/(.*)", "destination": "/index.html" }
  ]
}
```

_전체 설정 파일은 practice/chapter13/code/vercel.json 참고_

이 설정은 모든 경로 요청을 `index.html`로 리다이렉트한다. React Router가 클라이언트에서 라우팅을 처리할 수 있게 된다.

보안 헤더도 함께 설정하면 좋다:

```json
{
  "headers": [
    {
      "source": "/(.*)",
      "headers": [
        { "key": "X-Content-Type-Options", "value": "nosniff" },
        { "key": "X-Frame-Options", "value": "DENY" }
      ]
    }
  ]
}
```

## 13.3 환경 변수 관리

### 13.3.1 환경 변수의 필요성

개발 환경과 프로덕션 환경은 다른 설정을 사용한다. 예를 들어:

- **개발 환경**: 로컬 테스트용 Supabase 프로젝트
- **프로덕션 환경**: 실제 서비스용 Supabase 프로젝트

이런 환경별 설정을 코드에 직접 넣으면 문제가 생긴다:

```javascript
// ❌ 절대 금지: 하드코딩된 API 키
const supabase = createClient(
  'https://xxx.supabase.co',  // 환경마다 다름
  'eyJhbGciOiJIUzI1NiI...'    // 노출되면 보안 위험
);
```

**환경 변수**(Environment Variables)를 사용하면 설정을 코드에서 분리할 수 있다:

```javascript
// ✅ 권장: 환경 변수 사용
const supabase = createClient(
  import.meta.env.VITE_SUPABASE_URL,
  import.meta.env.VITE_SUPABASE_ANON_KEY
);
```

### 13.3.2 Vite 환경 변수 규칙

Vite에서 클라이언트(브라우저)에서 접근할 환경 변수는 반드시 `VITE_` 접두사를 붙여야 한다.

| 변수 이름 | 클라이언트 접근 | 용도 |
|-----------|-----------------|------|
| `VITE_SUPABASE_URL` | ✅ 가능 | Supabase 프로젝트 URL |
| `VITE_SUPABASE_ANON_KEY` | ✅ 가능 | 공개 가능한 익명 키 |
| `SUPABASE_SERVICE_KEY` | ❌ 불가 | 서버 전용 비밀 키 |

`VITE_` 접두사가 없는 변수는 빌드 시 번들에 포함되지 않는다. 이는 민감한 서버 전용 키가 클라이언트에 노출되는 것을 방지한다.

### 13.3.3 로컬 환경 설정

로컬 개발 시에는 `.env.local` 파일을 사용한다:

```bash
# .env.local (Git에서 제외)
VITE_SUPABASE_URL=https://your-project.supabase.co
VITE_SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...
```

_환경 변수 예시 파일은 practice/chapter13/code/13-3-env-example.txt 참고_

**.gitignore 필수 설정**:

```
.env
.env.local
.env.*.local
```

환경 변수 파일은 절대 Git에 커밋하면 안 된다. GitHub에 올라간 API 키는 봇에 의해 즉시 탐지되어 악용될 수 있다.

### 13.3.4 Vercel 환경 변수 설정

프로덕션 환경의 환경 변수는 Vercel 대시보드에서 설정한다.

**설정 방법**:

1. Vercel 대시보드 → 프로젝트 선택
2. Settings → Environment Variables
3. 변수 이름과 값 입력
4. 환경 선택 (Production / Preview / Development)
5. "Add" 버튼 클릭

```
┌─────────────────────────────────────────────┐
│ Environment Variables                        │
├─────────────────────────────────────────────┤
│ Name: VITE_SUPABASE_URL                     │
│ Value: https://prod-xxx.supabase.co         │
│                                             │
│ Environments:                               │
│ ☑ Production  ☑ Preview  ☐ Development     │
└─────────────────────────────────────────────┘
```

### 13.3.5 환경별 분리

Vercel은 세 가지 환경을 제공한다:

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│ Development │ ──▶ │   Preview   │ ──▶ │ Production  │
│  .env.local │     │  PR 브랜치   │     │ main 브랜치  │
│   로컬 개발  │     │  테스트 배포  │     │  실제 서비스  │
└─────────────┘     └─────────────┘     └─────────────┘
```

_환경 계층 다이어그램은 content/graphics/ch13/fig-13-2-env-hierarchy.mmd 참고_

각 환경에 다른 값을 설정할 수 있다:

| 환경 | Supabase 프로젝트 | 용도 |
|------|-------------------|------|
| Development | 개발용 프로젝트 | 로컬 테스트 |
| Preview | 스테이징 프로젝트 | PR 리뷰, QA |
| Production | 프로덕션 프로젝트 | 실제 사용자 서비스 |

**Vercel CLI로 환경 변수 관리**:

```bash
# 환경 변수 추가
vercel env add VITE_SUPABASE_URL

# 환경 변수 목록 조회
vercel env ls

# 로컬로 환경 변수 다운로드
vercel env pull .env.local
```

## 13.4 배포 후 확인

### 13.4.1 배포 URL 확인

배포가 완료되면 Vercel이 자동으로 URL을 생성한다:

| URL 유형 | 형식 | 설명 |
|----------|------|------|
| 프로젝트 URL | `project-name.vercel.app` | 기본 도메인 |
| 배포별 URL | `project-abc123.vercel.app` | 특정 배포 버전 |
| 프리뷰 URL | `project-git-branch.vercel.app` | PR/브랜치별 |

프로젝트 URL은 항상 최신 프로덕션 배포를 가리킨다. 배포별 URL은 해당 시점의 버전을 영구적으로 유지한다.

### 13.4.2 커스텀 도메인 연결

자신만의 도메인을 연결할 수 있다:

1. Vercel 대시보드 → Settings → Domains
2. 도메인 입력 (예: `myapp.com`)
3. DNS 설정 안내에 따라 레코드 추가
4. SSL 인증서 자동 발급 (몇 분 소요)

**DNS 설정 예시**:

| 타입 | 이름 | 값 |
|------|------|-----|
| A | @ | 76.76.21.21 |
| CNAME | www | cname.vercel-dns.com |

### 13.4.3 빌드 로그 확인

배포 실패 시 빌드 로그를 확인한다:

1. Vercel 대시보드 → Deployments
2. 실패한 배포 클릭
3. "Build Logs" 탭 확인

**흔한 빌드 오류와 해결책**:

| 오류 | 원인 | 해결책 |
|------|------|--------|
| `Module not found` | 의존성 누락 | `package.json` 확인 |
| `VITE_` 변수 undefined | 환경 변수 미설정 | Vercel 대시보드에서 설정 |
| `npm ERR! ERESOLVE` | 패키지 충돌 | `npm install --legacy-peer-deps` |
| 타입스크립트 오류 | 타입 에러 | 로컬에서 `npm run build` 확인 |

### 13.4.4 즉시 롤백

문제가 있는 배포를 이전 버전으로 되돌릴 수 있다. Vercel의 **Instant Rollback** 기능은 몇 초 만에 이전 상태로 복구한다.

**롤백 방법**:

1. Vercel 대시보드 → Deployments
2. 되돌아갈 정상 배포 선택
3. 점 세 개 메뉴(⋮) → "Promote to Production" 클릭

롤백은 새로운 배포를 생성하지 않고 기존 배포를 프로덕션으로 다시 지정한다. 따라서 수 초 내에 완료된다.

```
[현재 배포] ──문제 발생──▶ [롤백 실행] ──▶ [이전 배포 활성화]
    v3                                           v2
  (버그 있음)                                  (정상 동작)
```

### 13.4.5 Supabase URL 설정

배포된 앱이 Supabase 인증을 사용한다면, Supabase 설정도 업데이트해야 한다.

**Supabase 대시보드 설정**:

1. Authentication → URL Configuration
2. **Site URL**: Vercel 배포 URL로 변경
   - 예: `https://myapp.vercel.app`
3. **Redirect URLs**: 배포 URL 추가
   - 예: `https://myapp.vercel.app/**`

이 설정이 없으면 로그인 후 리다이렉트가 실패한다.

## 13.5 실습: 프로젝트 배포

### 13.5.1 배포 전 체크리스트

배포 전에 다음 항목을 확인한다:

**코드 준비**:
- [ ] `npm run build` 로컬에서 성공
- [ ] `npm run preview`로 빌드 결과 확인
- [ ] 콘솔 에러 없음

**환경 변수**:
- [ ] `.env` 파일이 `.gitignore`에 포함됨
- [ ] 모든 환경 변수가 `VITE_` 접두사 사용
- [ ] 민감한 키가 코드에 하드코딩되지 않음

**라우팅**:
- [ ] `vercel.json` 파일 생성 (React Router 사용 시)

**GitHub**:
- [ ] 최신 코드가 커밋됨
- [ ] main/master 브랜치에 푸시됨

_전체 체크리스트는 practice/chapter13/code/13-deploy-checklist.md 참고_

### 13.5.2 첫 배포 실행

**Step 1: 로컬 빌드 테스트**

```bash
npm run build
npm run preview
```

브라우저에서 `http://localhost:4173`에 접속하여 정상 동작 확인.

**Step 2: GitHub 푸시**

```bash
git add .
git commit -m "배포 준비: vercel.json 추가"
git push origin main
```

**Step 3: Vercel 프로젝트 생성**

1. [vercel.com/new](https://vercel.com/new) 접속
2. GitHub 저장소 선택 → Import
3. 환경 변수 설정:
   - `VITE_SUPABASE_URL`
   - `VITE_SUPABASE_ANON_KEY`
4. "Deploy" 클릭

**Step 4: 배포 확인**

배포가 완료되면 자동 생성된 URL로 접속:
- 홈페이지 로드 확인
- 로그인/로그아웃 테스트
- 게시글 CRUD 동작 확인

### 13.5.3 CI/CD 파이프라인 확인

이후 코드 변경 시 자동 배포가 작동하는지 확인한다:

1. 코드 수정 (예: 제목 변경)
2. `git commit && git push`
3. Vercel 대시보드에서 빌드 진행 확인
4. 배포 완료 후 변경사항 반영 확인

```
[개발자] ──git push──▶ [GitHub] ──웹훅──▶ [Vercel]
                                              │
                                    ┌─────────┴─────────┐
                                    ▼                   ▼
                              npm install          vite build
                                    │                   │
                                    └─────────┬─────────┘
                                              ▼
                                       CDN 배포
                                              │
                                              ▼
                                    배포 완료 (약 1분)
```

_CI/CD 파이프라인 다이어그램은 content/graphics/ch13/fig-13-3-cicd-pipeline.mmd 참고_

### 13.5.4 프리뷰 배포 활용

PR(Pull Request)을 생성하면 Vercel이 자동으로 프리뷰 환경을 생성한다.

**활용 시나리오**:

1. 새 기능 브랜치 생성: `git checkout -b feature/dark-mode`
2. 기능 구현 후 PR 생성
3. Vercel이 프리뷰 URL 생성: `project-git-feature-dark-mode.vercel.app`
4. 팀원이 프리뷰 URL에서 기능 검토
5. 승인 후 main에 머지 → 프로덕션 자동 배포

프리뷰 배포는 코드 리뷰를 훨씬 효과적으로 만든다. 리뷰어가 직접 기능을 테스트할 수 있기 때문이다.

## 13.6 트러블슈팅

### 13.6.1 빌드 실패

**증상**: Vercel 빌드가 실패하고 에러 메시지 표시

**해결 순서**:

1. Vercel 빌드 로그에서 정확한 에러 확인
2. 로컬에서 `npm run build` 실행하여 동일 에러 재현
3. 에러 수정 후 다시 푸시

**Node.js 버전 문제**:

Vercel 기본 Node.js 버전과 로컬이 다를 수 있다. `package.json`에 명시:

```json
{
  "engines": {
    "node": ">=18.0.0"
  }
}
```

또는 Vercel 대시보드 → Settings → General → Node.js Version에서 설정.

### 13.6.2 환경 변수 문제

**증상**: `import.meta.env.VITE_*`가 undefined

**확인 사항**:

1. 변수 이름에 `VITE_` 접두사가 있는가?
2. Vercel 대시보드에 변수가 등록되어 있는가?
3. 환경 변수 변경 후 재배포했는가?

환경 변수를 변경하면 반드시 재배포가 필요하다. Vercel 대시보드에서 "Redeploy" 클릭.

### 13.6.3 라우팅 404 에러

**증상**: 직접 URL 접속 시 404 페이지 표시

**해결**: `vercel.json` 파일 확인

```json
{
  "rewrites": [
    { "source": "/(.*)", "destination": "/index.html" }
  ]
}
```

파일이 프로젝트 루트에 있어야 하고, GitHub에 커밋되어 있어야 한다.

### 13.6.4 CORS 에러

**증상**: API 호출 시 CORS 에러 발생

**원인**: Supabase에 배포 URL이 등록되지 않음

**해결**:

1. Supabase 대시보드 → Authentication → URL Configuration
2. Redirect URLs에 Vercel 배포 URL 추가
3. 와일드카드 사용: `https://myapp.vercel.app/**`

## 장 요약

- **서버리스 호스팅**은 서버 관리 없이 코드만으로 배포하는 방식이다
- **Vercel**은 프론트엔드에 최적화된 배포 플랫폼으로, GitHub 연동을 통한 자동 배포를 제공한다
- **CI/CD 파이프라인**은 코드 푸시 → 빌드 → 배포를 자동화한다
- **환경 변수**는 `VITE_` 접두사를 사용하고, Vercel 대시보드에서 설정한다
- **프리뷰 배포**는 PR마다 별도 URL을 생성하여 코드 리뷰를 돕는다
- **즉시 롤백**으로 문제 발생 시 수 초 내에 이전 버전으로 복구할 수 있다
- SPA 라우팅은 **vercel.json**의 rewrites 설정으로 해결한다
