# 14장 프로젝트 설계

## 학습목표

1. 유저 스토리와 기능 명세서를 작성할 수 있다
2. ERD를 활용하여 데이터 모델을 설계할 수 있다
3. 역할과 권한 매트릭스를 정의할 수 있다
4. AI 협업을 위한 작업 분해와 검증 계획을 수립할 수 있다

## 14.1 요구사항 정의

### 14.1.1 유저 스토리 작성법

프로젝트의 첫 단계는 "무엇을 만들 것인가"를 명확히 하는 것이다. **유저 스토리**(User Story)는 사용자 관점에서 기능을 정의하는 방법이다.

유저 스토리는 다음 형식을 따른다:

```
"<역할>로서, <기능>을 원한다, 왜냐하면 <가치>이기 때문이다"
```

영문 형식:

```
"As a <role>, I want <feature>, so that <benefit>"
```

예를 들어, 게시판의 글 작성 기능은 이렇게 표현한다:

> "회원으로서, 게시글을 작성하고 싶다, 왜냐하면 내 의견을 공유하고 싶기 때문이다"

유저 스토리의 핵심은 **기술적 구현**이 아니라 **사용자 가치**를 설명하는 것이다. "데이터베이스에 INSERT한다"가 아니라 "내 의견을 공유한다"로 표현한다.

### 에픽과 스토리 계층

큰 기능은 **에픽**(Epic)으로 묶고, 세부 기능을 스토리로 분해한다:

```
[에픽] 게시판 기능
├── [스토리] 게시글 목록 조회
├── [스토리] 게시글 상세 보기
├── [스토리] 게시글 작성
├── [스토리] 게시글 수정
└── [스토리] 게시글 삭제
```

### 인수 조건

각 스토리에는 **인수 조건**(Acceptance Criteria)을 명시한다. 인수 조건은 "이 스토리가 완료되었다"고 판단하는 기준이다.

**Given-When-Then** 형식을 권장한다:

| 요소 | 설명 | 예시 |
|------|------|------|
| **Given** | 전제 조건 | 로그인한 상태에서 |
| **When** | 사용자 행동 | 게시글 작성 버튼을 클릭하면 |
| **Then** | 기대 결과 | 작성 폼이 표시된다 |

```markdown
스토리: 게시글 작성
인수 조건:
- Given: 로그인한 상태에서
- When: 제목과 내용을 입력하고 등록 버튼을 클릭하면
- Then: 게시글이 저장되고 목록에 표시된다
- Then: 작성자 정보가 자동으로 기록된다
```

_유저 스토리 구조 다이어그램은 content/graphics/ch14/fig-14-1-user-story-format.mmd 참고_

### 14.1.2 기능 명세서 템플릿

유저 스토리를 더 상세하게 문서화한 것이 **기능 명세서**다. 개발 시 참조할 수 있도록 기술적 세부사항을 포함한다.

**기능 명세서 항목**:

| 항목 | 설명 |
|------|------|
| **기능 ID** | 고유 식별자 (예: US-001) |
| **기능명** | 기능의 이름 |
| **설명** | 기능의 상세 설명 |
| **선행 조건** | 기능 실행에 필요한 조건 |
| **입력** | 필요한 데이터 |
| **출력** | 결과 데이터/화면 |
| **예외 케이스** | 오류 상황과 처리 방법 |
| **우선순위** | MoSCoW 분류 |

### MoSCoW 우선순위

**MoSCoW**는 기능의 우선순위를 분류하는 방법이다:

| 분류 | 의미 | 권장 비율 |
|------|------|-----------|
| **M**ust Have | 필수, 없으면 출시 불가 | ~60% |
| **S**hould Have | 중요하지만 연기 가능 | ~20% |
| **C**ould Have | 있으면 좋음 | ~20% |
| **W**on't Have | 이번에는 제외 | - |

**MVP**(Minimum Viable Product, 최소 기능 제품)는 Must Have만으로 구성한다. Should Have와 Could Have는 시간이 허락할 때 추가한다.

**게시판 기능 MoSCoW 예시**:

| 분류 | 기능 |
|------|------|
| **Must Have** | 목록 조회, 상세 보기, 작성, 로그인 |
| **Should Have** | 수정, 삭제, 페이지네이션 |
| **Could Have** | 댓글, 검색, 좋아요 |
| **Won't Have** | 파일 첨부, 알림, 태그 |

## 14.2 데이터 모델 설계

### 14.2.1 ERD 작성

**ERD**(Entity-Relationship Diagram)는 데이터베이스 구조를 시각화한 다이어그램이다. 테이블을 만들기 전에 ERD로 설계하면 실수를 줄일 수 있다.

ERD의 주요 구성 요소:

| 요소 | 표현 | 설명 |
|------|------|------|
| **엔티티** | 사각형 | 데이터를 저장할 대상 (User, Post) |
| **속성** | 엔티티 내 목록 | 엔티티의 특성 (name, email) |
| **관계** | 선 | 엔티티 간 연결 |
| **카디널리티** | 기호 | 관계의 수량 (1:N, N:M) |

### 카디널리티 표기

엔티티 간 관계의 수량을 표현한다:

| 표기 | 의미 | 예시 |
|------|------|------|
| 1:1 | 일대일 | 사용자 - 프로필 |
| 1:N | 일대다 | 사용자 - 게시글 |
| N:M | 다대다 | 게시글 - 태그 |

게시판의 관계를 분석해보자:

- 사용자 한 명이 여러 게시글을 작성한다 → **1:N**
- 사용자 한 명이 여러 댓글을 작성한다 → **1:N**
- 게시글 하나에 여러 댓글이 달린다 → **1:N**

### 게시판 ERD

```
┌─────────────────┐
│     users       │
├─────────────────┤
│ id (PK)         │
│ email           │
│ name            │
│ created_at      │
└────────┬────────┘
         │ 1
         │
         │ N
┌────────▼────────┐       ┌─────────────────┐
│     posts       │       │    comments     │
├─────────────────┤       ├─────────────────┤
│ id (PK)         │       │ id (PK)         │
│ author_id (FK)  │◄──────│ post_id (FK)    │
│ title           │  1  N │ author_id (FK)  │
│ content         │       │ content         │
│ created_at      │       │ created_at      │
└─────────────────┘       └─────────────────┘
```

### 14.2.2 테이블 스키마 정의

ERD를 실제 테이블로 변환한다. 각 컬럼의 자료형과 제약조건을 정의한다.

**게시글 테이블 스키마**:

| 컬럼명 | 자료형 | 제약조건 | 설명 |
|--------|--------|----------|------|
| id | UUID | PK, DEFAULT gen_random_uuid() | 고유 식별자 |
| author_id | UUID | FK → auth.users(id), NOT NULL | 작성자 |
| title | VARCHAR(200) | NOT NULL | 제목 |
| content | TEXT | NOT NULL | 본문 |
| view_count | INTEGER | DEFAULT 0 | 조회수 |
| created_at | TIMESTAMPTZ | DEFAULT NOW() | 작성일 |
| updated_at | TIMESTAMPTZ | DEFAULT NOW() | 수정일 |

**자료형 선택 기준**:

| 자료형 | 용도 | 예시 |
|--------|------|------|
| UUID | 고유 식별자 | id, author_id |
| VARCHAR(n) | 길이 제한 텍스트 | title, username |
| TEXT | 긴 텍스트 | content, description |
| INTEGER | 숫자 | view_count, order |
| BOOLEAN | 참/거짓 | is_published, is_admin |
| TIMESTAMPTZ | 시간 | created_at, updated_at |

**제약조건**:

| 제약조건 | 의미 |
|----------|------|
| PRIMARY KEY (PK) | 테이블 내 고유 식별 |
| FOREIGN KEY (FK) | 다른 테이블 참조 |
| NOT NULL | 빈 값 불허 |
| UNIQUE | 중복 불허 |
| DEFAULT | 기본값 설정 |
| CHECK | 조건 검증 |

**Supabase SQL 예시**:

```sql
CREATE TABLE posts (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    author_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
    title VARCHAR(200) NOT NULL,
    content TEXT NOT NULL,
    view_count INTEGER DEFAULT 0,
    created_at TIMESTAMPTZ DEFAULT NOW(),
    updated_at TIMESTAMPTZ DEFAULT NOW()
);
```

`ON DELETE CASCADE`는 사용자가 삭제되면 해당 사용자의 게시글도 함께 삭제된다는 의미다.

_전체 스키마 SQL은 practice/chapter14/code/14-2-schema-design.sql 참고_

## 14.3 권한 규칙 설계

### 14.3.1 역할 정의

11장에서 배운 RLS를 적용하려면 먼저 **역할**(Role)을 정의해야 한다. 게시판에서 일반적인 역할 구분은 다음과 같다:

| 역할 | 설명 | 식별 방법 |
|------|------|-----------|
| **Anonymous** | 비로그인 사용자 | `auth.uid() IS NULL` |
| **Member** | 일반 회원 | `auth.uid() IS NOT NULL` |
| **Admin** | 관리자 | `profiles.role = 'admin'` |

관리자 역할을 구분하려면 프로필 테이블에 `role` 컬럼을 추가한다:

```sql
CREATE TABLE profiles (
    id UUID PRIMARY KEY REFERENCES auth.users(id),
    username VARCHAR(50) NOT NULL,
    role VARCHAR(20) DEFAULT 'member' CHECK (role IN ('member', 'admin'))
);
```

### 14.3.2 권한 매트릭스 작성

**권한 매트릭스**는 "누가 무엇을 할 수 있는가"를 표로 정리한 것이다. CRUD 각 작업에 대해 역할별 권한을 명시한다.

**게시글 권한 매트릭스**:

| 작업 | Anonymous | Member | Admin |
|------|-----------|--------|-------|
| Create | ✗ | ✓ | ✓ |
| Read | ✓ | ✓ | ✓ |
| Update | ✗ | 본인만 | ✓ |
| Delete | ✗ | 본인만 | ✓ |

이 매트릭스를 RLS 정책으로 변환한다:

**읽기 정책 (누구나)**:

```sql
CREATE POLICY "posts_read_all" ON posts
    FOR SELECT USING (true);
```

**생성 정책 (로그인 사용자)**:

```sql
CREATE POLICY "posts_create_auth" ON posts
    FOR INSERT WITH CHECK (auth.uid() IS NOT NULL);
```

**수정 정책 (작성자 또는 관리자)**:

```sql
CREATE POLICY "posts_update_owner" ON posts
    FOR UPDATE USING (
        auth.uid() = author_id
        OR EXISTS (
            SELECT 1 FROM profiles
            WHERE id = auth.uid() AND role = 'admin'
        )
    );
```

### 권한 설계 체크리스트

권한 설계 시 다음을 확인한다:

- [ ] 모든 테이블에 RLS가 활성화되었는가?
- [ ] CRUD 각 작업에 정책이 있는가?
- [ ] 권한 상승 경로가 없는가? (Member → Admin)
- [ ] 다른 사용자 데이터 접근이 차단되는가?
- [ ] 관리자 기능이 올바르게 동작하는가?

## 14.4 AI 협업 계획

### 14.4.1 작업 분해

프로젝트를 성공적으로 완료하려면 작업을 적절한 크기로 분해해야 한다. **WBS**(Work Breakdown Structure)는 프로젝트를 단계와 작업으로 나누는 방법이다.

**게시판 프로젝트 WBS 예시**:

```
게시판 프로젝트
├── Phase 1: 환경 설정
│   ├── 프로젝트 생성 (Vite)
│   ├── ESLint/Prettier 설정
│   └── Supabase 클라이언트 설정
├── Phase 2: 인증 구현
│   ├── AuthContext 작성
│   ├── 로그인 버튼 컴포넌트
│   └── 보호된 라우트
├── Phase 3: CRUD 구현
│   ├── 게시글 목록
│   ├── 게시글 상세
│   ├── 게시글 작성
│   └── 수정/삭제
├── Phase 4: 보안 및 UX
│   ├── RLS 정책 작성
│   └── 에러/로딩 처리
└── Phase 5: 배포
    ├── Vercel 배포
    └── 환경 변수 설정
```

각 작업은 **하루 이내**에 완료할 수 있는 크기로 분해한다.

### 14.4.2 AI에게 위임할 작업 선정

모든 작업을 AI에게 맡기면 안 된다. 작업의 특성에 따라 AI와 인간의 역할을 구분한다.

**AI에게 위임하기 좋은 작업**:

| 작업 유형 | 예시 | 이유 |
|-----------|------|------|
| **보일러플레이트** | 컴포넌트 구조, import | 반복적, 패턴화 |
| **반복 코드** | CRUD 함수 | 정형화된 로직 |
| **스타일링** | CSS, Tailwind | 창의성 불필요 |
| **유틸리티** | 날짜 포맷, 에러 메시지 | 표준 패턴 존재 |
| **문서화** | 주석, README | 코드 기반 생성 |

**인간이 반드시 수행할 작업**:

| 작업 유형 | 이유 |
|-----------|------|
| **보안 정책** | 잘못된 RLS는 데이터 유출 위험 |
| **API 키 관리** | 시크릿 노출 방지 |
| **아키텍처 결정** | 프로젝트 방향성 결정 |
| **최종 코드 리뷰** | AI 출력물 검증 필수 |
| **배포 작업** | 프로덕션 환경 신중히 |

_AI 협업 역할 다이어그램은 content/graphics/ch14/fig-14-3-ai-collaboration.mmd 참고_

### AI 협업 비율 가이드

| 프로젝트 단계 | AI 비율 | 인간 비율 |
|---------------|---------|-----------|
| 환경 설정 | 70% | 30% |
| 인증 구현 | 50% | 50% |
| CRUD 구현 | 60% | 40% |
| 보안 적용 | 20% | 80% |
| 배포 | 10% | 90% |

보안과 배포는 인간의 검증이 특히 중요하다.

### 14.4.3 검증 포인트 설정

AI가 생성한 코드는 반드시 검증해야 한다. 단계별 **체크포인트**를 설정한다.

**코드 생성 후 검증 체크리스트**:

```markdown
□ 버전 호환성
  - 사용된 API가 현재 라이브러리 버전과 맞는가?
  - deprecated 경고가 없는가?

□ 보안 검토
  - 하드코딩된 시크릿이 없는가?
  - XSS 취약점이 없는가?

□ 스타일 일관성
  - 기존 코드 컨벤션을 따르는가?
  - import 경로가 올바른가?

□ 실제 동작 확인
  - 브라우저에서 실행해 보았는가?
  - 콘솔 에러가 없는가?
```

**단계별 리뷰 포인트**:

| 단계 | 리뷰 항목 |
|------|-----------|
| 환경 설정 후 | 개발 서버 정상 실행 확인 |
| 인증 구현 후 | 로그인/로그아웃 플로우 테스트 |
| CRUD 구현 후 | 모든 작업 정상 동작 확인 |
| 보안 적용 후 | 권한 없는 접근 차단 확인 |
| 배포 후 | 프로덕션 환경 기능 테스트 |

### AI 프롬프트 작성 팁

AI에게 작업을 요청할 때 다음 정보를 포함한다:

```
React 18, Supabase JS v2 환경에서
{컴포넌트명} 컴포넌트를 작성해줘.

요구사항:
- 함수형 컴포넌트
- {기능 설명}
- 에러 처리 포함
- 로딩 상태 포함

참고: src/lib/supabase.js에서 supabase 클라이언트 import
```

버전을 명시하면 AI가 올바른 API를 사용할 가능성이 높아진다.

- **인수 조건**은 Given-When-Then 형식으로 완료 기준을 명확히 한다
- **MoSCoW**는 Must/Should/Could/Won't로 기능 우선순위를 분류한다
- **ERD**로 엔티티와 관계를 시각화한 후 테이블 스키마를 정의한다
- **권한 매트릭스**는 역할별 CRUD 권한을 표로 정리하고 RLS 정책으로 변환한다
- AI 협업 시 **보일러플레이트는 AI**, **보안은 인간**이 담당한다
- **검증 체크리스트**로 AI 출력물을 단계별로 검증한다
