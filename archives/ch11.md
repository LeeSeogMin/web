## 11.1 RLS의 필요성

### 11.1.1 클라이언트 사이드 보안의 한계

10장에서 구현한 게시판에는 문제가 있다. "본인 글만 수정/삭제" 조건을 프론트엔드에서만 체크하고 있다:

```jsx
// PostDetail.jsx
const isAuthor = user?.id === post.user_id;

{isAuthor && (
    <button onClick={handleDelete}>삭제</button>
)}
```

이 코드는 **버튼을 숨길 뿐**, 실제 삭제를 막지 못한다. 악의적인 사용자가 브라우저 개발자 도구에서 다음과 같이 호출하면?

```javascript
// 개발자 도구 콘솔에서
await supabase.from('posts').delete().eq('id', '타인의-게시글-id');
```

Supabase API 키는 브라우저에 노출되어 있으므로, **누구나 API를 직접 호출**할 수 있다.

### 11.1.2 서버 사이드 권한 강제

**Row Level Security(RLS)**는 PostgreSQL의 기능으로, **데이터베이스 레벨**에서 접근 권한을 강제한다.

RLS를 사용하면:
- 쿼리가 실행되기 전에 정책 조건을 검사
- 조건을 만족하지 않는 행은 결과에서 제외 또는 에러 발생
- 프론트엔드를 우회해도 **데이터베이스가 막아줌**

### 11.1.3 RLS란 무엇인가

**RLS(Row Level Security)**는 테이블의 **행(row) 단위**로 접근을 제어하는 보안 기능이다.

**비유**: RLS는 건물의 출입 카드 시스템과 같다.
- 건물 입구(API)는 누구나 접근 가능
- 하지만 각 방(테이블 행)은 권한이 있는 사람만 입장
- 권한 카드(JWT)로 신원을 확인하고, 정책에 따라 출입 허용/거부

**표 11.1** 프론트엔드 보안 vs RLS

| 항목 | 프론트엔드 보안 | RLS |
|------|----------------|-----|
| 실행 위치 | 브라우저 | 데이터베이스 |
| 우회 가능성 | 가능 (개발자 도구) | 불가능 |
| 역할 | UX 개선 | 실제 보안 |
| 결론 | 보조 수단 | 필수 |

> **중요**: 프론트엔드 권한 체크와 RLS는 **둘 다 필요**하다. 프론트엔드는 좋은 UX를 위해, RLS는 실제 보안을 위해 사용한다.

## 11.2 RLS 기본 문법

### 11.2.1 RLS 활성화

RLS를 사용하려면 테이블별로 활성화해야 한다:

```sql
-- RLS 활성화
ALTER TABLE posts ENABLE ROW LEVEL SECURITY;

-- RLS 비활성화 (개발 중에만, 프로덕션 금지)
ALTER TABLE posts DISABLE ROW LEVEL SECURITY;
```

> **주의**: RLS를 활성화하면 정책을 정의하기 전까지 **모든 접근이 차단**된다.

Supabase Table Editor에서 테이블을 생성하면 RLS가 **기본 활성화**되어 있다. SQL Editor에서 직접 생성한 테이블은 수동으로 활성화해야 한다.

### 11.2.2 정책 생성: CREATE POLICY

**정책(Policy)**은 "누가, 어떤 작업을, 어떤 조건에서 할 수 있는가"를 정의한다.

```sql
CREATE POLICY policy_name    -- 정책 이름 (테이블 내 고유)
ON table_name                -- 대상 테이블
FOR operation                -- 작업: ALL, SELECT, INSERT, UPDATE, DELETE
TO role_name                 -- 역할: authenticated, anon, public
USING (condition)            -- 기존 행 필터링 조건
WITH CHECK (condition);      -- 새 행 검증 조건
```

**예시 - 모든 사용자가 게시글을 읽을 수 있음**:
```sql
CREATE POLICY "Anyone can read posts"
ON posts
FOR SELECT
TO public
USING ( true );
```

### 11.2.3 USING과 WITH CHECK 차이

```
┌─────────────────────────────────────────────────────────────┐
│                  USING vs WITH CHECK                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   USING 조건 (기존 행)          WITH CHECK 조건 (새 행)     │
│   ┌────────────────┐           ┌────────────────┐          │
│   │   데이터베이스  │           │    새 데이터   │          │
│   │   기존 행들    │           │   (INSERT/     │          │
│   └───────┬────────┘           │    UPDATE)     │          │
│           │                    └───────┬────────┘          │
│           ▼                            ▼                   │
│   ┌────────────────┐           ┌────────────────┐          │
│   │   조건 검사    │           │   조건 검사    │          │
│   └───────┬────────┘           └───────┬────────┘          │
│     ┌─────┴─────┐                ┌─────┴─────┐             │
│     ▼           ▼                ▼           ▼             │
│  [true]     [false]           [true]     [false]           │
│  행 반환    행 숨김           저장 허용   에러 발생         │
│                                                             │
│   사용 연산:                   사용 연산:                   │
│   - SELECT                     - INSERT                     │
│   - UPDATE (기존 값)           - UPDATE (새 값)             │
│   - DELETE                                                  │
│                                                             │
└─────────────────────────────────────────────────────────────┘

※ 그림 11.2 USING vs WITH CHECK
```

**표 11.2** 연산별 USING/WITH CHECK 사용

| 연산 | USING | WITH CHECK | 설명 |
|------|-------|------------|------|
| SELECT | O | X | 기존 행 필터링 |
| INSERT | X | O | 새 행 검증 |
| UPDATE | O | O | 기존 행 + 새 값 모두 검증 |
| DELETE | O | X | 기존 행 필터링 |

**기본 동작**: WITH CHECK를 생략하면 USING 조건이 WITH CHECK에도 적용된다.

### 11.2.4 auth.uid() 함수

`auth.uid()`는 Supabase가 제공하는 함수로, **현재 로그인한 사용자의 UUID**를 반환한다.

```sql
-- 본인 데이터만 조회
USING ( auth.uid() = user_id )
```

**반환값**:
- 로그인한 사용자: 해당 사용자의 UUID
- 로그인하지 않은 사용자: NULL

**성능 최적화**: 서브쿼리로 감싸면 쿼리당 한 번만 실행되어 성능이 향상된다.

```sql
-- 비권장 (행마다 함수 호출)
USING ( auth.uid() = user_id )

-- 권장 (결과 캐싱)
USING ( (select auth.uid()) = user_id )
```

### 11.2.5 역할(Role)

**TO** 절에 사용하는 역할:

**표 11.3** Supabase 역할

| 역할 | 설명 |
|------|------|
| `public` | 모든 요청 (anon + authenticated) |
| `anon` | 인증되지 않은 요청 |
| `authenticated` | 인증된 사용자 요청 |
| `service_role` | RLS 우회하는 관리자 역할 |

`TO authenticated`를 사용하면 로그인하지 않은 사용자는 정책 조건을 검사하기도 전에 차단된다.

## 11.3 권한 시나리오 구현

### 11.3.1 누구나 읽기 가능

게시글은 로그인 여부와 관계없이 누구나 볼 수 있어야 한다.

```sql
CREATE POLICY "Anyone can read posts"
ON posts
FOR SELECT
TO public
USING ( true );  -- 모든 행 허용
```

**USING ( true )**는 모든 행에 대해 조건이 참이므로, 모든 게시글이 조회 가능하다.

### 11.3.2 로그인 사용자만 작성

게시글 작성은 로그인한 사용자만 가능하고, user_id가 본인 id와 일치해야 한다.

```sql
CREATE POLICY "Authenticated can insert posts"
ON posts
FOR INSERT
TO authenticated  -- 로그인한 사용자만
WITH CHECK ( (select auth.uid()) = user_id );  -- user_id가 본인
```

이 정책은 두 가지를 검증한다:
1. `TO authenticated`: 로그인 여부
2. `WITH CHECK`: user_id가 본인 id인지

악의적인 사용자가 다른 사람의 user_id로 게시글을 작성하려 해도 WITH CHECK에서 막힌다.

### 11.3.3 작성자만 수정/삭제

**수정 정책**:
```sql
CREATE POLICY "Owner can update posts"
ON posts
FOR UPDATE
TO authenticated
USING ( (select auth.uid()) = user_id )      -- 기존 행: 본인 글만 선택
WITH CHECK ( (select auth.uid()) = user_id ); -- 새 값: user_id 변경 방지
```

UPDATE는 USING과 WITH CHECK 둘 다 필요하다:
- **USING**: 수정 대상 행이 본인 것인지 확인
- **WITH CHECK**: 수정 후에도 user_id가 본인인지 확인 (소유권 이전 방지)

**삭제 정책**:
```sql
CREATE POLICY "Owner can delete posts"
ON posts
FOR DELETE
TO authenticated
USING ( (select auth.uid()) = user_id );
```

DELETE는 새 데이터가 없으므로 USING만 사용한다.

### 11.3.4 관리자 전체 권한

관리자는 모든 게시글에 대해 CRUD가 가능해야 한다.

**1단계: 관리자 확인 함수 생성**
```sql
CREATE OR REPLACE FUNCTION is_admin()
RETURNS boolean
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = public
AS $$
BEGIN
    RETURN (
        SELECT COALESCE(is_admin, false)
        FROM profiles
        WHERE id = auth.uid()
    );
END;
$$;
```

**2단계: 관리자 정책 추가**
```sql
CREATE POLICY "Admin full access"
ON posts
FOR ALL
TO authenticated
USING ( (select is_admin()) )
WITH CHECK ( (select is_admin()) );
```

> **팁**: 여러 정책이 있을 때 **OR 조건**처럼 작동한다. 작성자 정책과 관리자 정책 중 하나라도 통과하면 접근 가능하다.

## 11.4 RLS 테스트와 디버깅

### 11.4.1 SQL Editor에서 테스트

Supabase 대시보드의 SQL Editor에서 역할을 전환하여 테스트할 수 있다.

```sql
-- 특정 사용자로 테스트
SET request.jwt.claim.sub = 'user-uuid-here';

-- anon 역할로 테스트
SET role anon;

-- authenticated 역할로 테스트
SET role authenticated;

-- 원래대로 복원
RESET role;
```

### 11.4.2 권한 실패 케이스

**SELECT 실패**: 에러 없이 해당 행이 결과에서 제외된다.
```javascript
// 타인의 비공개 게시글 조회 시도
const { data } = await supabase.from('posts').select('*');
// data에 타인의 비공개 글이 없음 (에러 없음)
```

**INSERT/UPDATE/DELETE 실패**: 에러가 발생하거나 영향받는 행이 0개다.
```javascript
// 타인의 게시글 삭제 시도
const { error } = await supabase.from('posts').delete().eq('id', 'others-post-id');
// error 또는 영향받는 행 0개
```

**표 11.4** 권한 실패 시 동작

| 연산 | 정책 실패 시 |
|------|-------------|
| SELECT | 해당 행 제외 (조용히) |
| INSERT | 에러 발생 |
| UPDATE | 영향받는 행 0개 또는 에러 |
| DELETE | 영향받는 행 0개 |

### 11.4.3 정책 충돌 해결

**같은 연산에 여러 정책이 있으면 OR 조건**으로 동작한다:

```sql
-- 정책 1: 본인 글만
CREATE POLICY "Owner access" ON posts FOR SELECT
USING ( auth.uid() = user_id );

-- 정책 2: 공개 글은 모두
CREATE POLICY "Public posts" ON posts FOR SELECT
USING ( is_public = true );

-- 결과: (본인 글) OR (공개 글) 모두 조회 가능
```

**정책 확인 쿼리**:
```sql
SELECT schemaname, tablename, policyname, permissive, roles, cmd
FROM pg_policies
WHERE schemaname = 'public'
ORDER BY tablename;
```

**정책 삭제**:
```sql
DROP POLICY IF EXISTS "policy_name" ON table_name;
```

### 11.4.4 자주 하는 실수

1. **RLS 활성화 후 정책 없음**: 모든 접근 차단됨
2. **UPDATE에 SELECT 정책 없음**: UPDATE 전에 행을 읽어야 하므로 SELECT 정책도 필요
3. **USING (true) 남발**: 모든 행 허용하므로 보안 취약
4. **auth.uid() 서브쿼리 미사용**: 성능 저하

## 11.5 실습: 게시판 권한 정책

### 요구사항

10장에서 만든 게시판에 다음 권한을 적용한다:

**posts 테이블**:
- 누구나 조회 가능
- 로그인 사용자만 작성 가능 (user_id = 본인)
- 작성자만 수정/삭제 가능

**comments 테이블**:
- 누구나 조회 가능
- 로그인 사용자만 작성 가능
- 작성자만 삭제 가능

**profiles 테이블**:
- 누구나 조회 가능
- 본인만 수정 가능

### 전체 SQL

```sql
-- 기존 정책 삭제
DROP POLICY IF EXISTS "posts_select_policy" ON posts;
DROP POLICY IF EXISTS "posts_insert_policy" ON posts;
DROP POLICY IF EXISTS "posts_update_policy" ON posts;
DROP POLICY IF EXISTS "posts_delete_policy" ON posts;

-- RLS 활성화
ALTER TABLE posts ENABLE ROW LEVEL SECURITY;
ALTER TABLE comments ENABLE ROW LEVEL SECURITY;
ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;

-- posts 정책
CREATE POLICY "posts_select_policy"
ON posts FOR SELECT TO public
USING ( true );

CREATE POLICY "posts_insert_policy"
ON posts FOR INSERT TO authenticated
WITH CHECK ( (select auth.uid()) = user_id );

CREATE POLICY "posts_update_policy"
ON posts FOR UPDATE TO authenticated
USING ( (select auth.uid()) = user_id )
WITH CHECK ( (select auth.uid()) = user_id );

CREATE POLICY "posts_delete_policy"
ON posts FOR DELETE TO authenticated
USING ( (select auth.uid()) = user_id );

-- comments 정책
CREATE POLICY "comments_select_policy"
ON comments FOR SELECT TO public
USING ( true );

CREATE POLICY "comments_insert_policy"
ON comments FOR INSERT TO authenticated
WITH CHECK ( (select auth.uid()) = user_id );

CREATE POLICY "comments_delete_policy"
ON comments FOR DELETE TO authenticated
USING ( (select auth.uid()) = user_id );

-- profiles 정책
CREATE POLICY "profiles_select_policy"
ON profiles FOR SELECT TO public
USING ( true );

CREATE POLICY "profiles_update_policy"
ON profiles FOR UPDATE TO authenticated
USING ( (select auth.uid()) = id )
WITH CHECK ( (select auth.uid()) = id );
```

### 테스트 시나리오

1. **로그인하지 않은 사용자**:
   - 게시글 조회: 성공
   - 게시글 작성: 실패 (에러)

2. **로그인한 사용자 A**:
   - 본인 게시글 수정: 성공
   - 타인 게시글 수정: 실패 (영향받는 행 0개)

3. **삭제 테스트**:
   - 본인 게시글 삭제: 성공
   - 타인 게시글 삭제: 실패

## 핵심 정리

1. **클라이언트 사이드 보안은 우회 가능**하다. 프론트엔드 권한 체크는 UX를 위한 것이고, 실제 보안은 RLS에서 처리해야 한다.

2. **RLS**는 PostgreSQL의 행 단위 접근 제어 기능이다. 테이블에 정책을 정의하여 "누가, 어떤 행에, 어떤 작업을" 할 수 있는지 제어한다.

3. **USING**은 기존 행을 필터링하고(SELECT, UPDATE 대상, DELETE), **WITH CHECK**는 새 데이터를 검증한다(INSERT, UPDATE 새 값).

4. **auth.uid()**는 현재 로그인한 사용자의 UUID를 반환한다. 성능을 위해 `(select auth.uid())` 형태로 사용한다.

5. **TO 역할**: `public`(모두), `anon`(미인증), `authenticated`(인증), `service_role`(RLS 우회)

6. 같은 연산에 **여러 정책은 OR 조건**으로 동작한다. 하나라도 통과하면 접근 가능.

7. **테스트 방법**: SQL Editor에서 역할을 전환하거나, 다른 브라우저에서 다른 계정으로 테스트한다.

## 연습문제

### 기초

**문제 1.** 프론트엔드에서 권한을 체크하는 것만으로는 보안이 충분하지 않은 이유를 설명하시오.

**문제 2.** USING과 WITH CHECK의 차이를 설명하고, UPDATE 연산에서 둘 다 필요한 이유를 서술하시오.

**문제 3.** auth.uid() 함수의 역할을 설명하고, 로그인하지 않은 사용자에게 어떤 값을 반환하는지 쓰시오.

### 중급

**문제 4.** comments 테이블에 다음 정책을 SQL로 작성하시오:
- 누구나 조회 가능
- 로그인 사용자만 작성 가능
- 작성자 또는 해당 게시글 작성자가 삭제 가능

**문제 5.** "비공개 게시글"을 구현하려 한다. posts 테이블에 `is_public` 열(boolean)을 추가했을 때, "공개 글은 누구나, 비공개 글은 작성자만 조회 가능"한 SELECT 정책을 작성하시오.

### 심화

**문제 6.** 역할 기반 권한 시스템을 설계하시오. profiles 테이블에 `role` 열('user', 'moderator', 'admin')을 추가하고:
- user: 본인 글만 수정/삭제
- moderator: 모든 글 수정 가능, 삭제 불가
- admin: 모든 권한

이를 위한 함수와 정책을 작성하시오.
