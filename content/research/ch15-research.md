# 15장 리서치: 프로젝트 구현과 검증

## 1. Pull Request 베스트 프랙티스

### PR 작성 원칙
- **작고 집중된 PR**: 단일 목적을 수행하는 작은 PR
- **명확한 제목과 설명**: 리뷰어가 빠르게 이해할 수 있도록
- **자기 리뷰 먼저**: 제출 전 직접 검토, 빌드, 테스트
- **Draft PR 활용**: 진행 중인 작업에 피드백 받기

### 리뷰 수행 원칙
- **신속한 리뷰**: 제출 후 2시간 이내 리뷰 시작 권장
- **코드에 집중**: 사람이 아닌 코드를 비평
- **건설적 피드백**: 개인 공격이 아닌 개선 제안
- **자동화 활용**: 스타일 체크는 자동화, 리뷰어는 로직에 집중

### 시각적 데모
- 스크린샷, 화면 캡처, GIF 첨부 권장
- GitHub은 동영상과 GIF 모두 지원
- 변경사항을 시각적으로 보여주기

## 2. README 베스트 프랙티스

### 필수 섹션
1. **프로젝트 소개**: 무엇을 하는지, 왜 중요한지
2. **Quick Start**: 10분 내 설치/실행 가능하도록
3. **디렉토리 구조**: tree 명령어로 자동 생성 가능
4. **사용 예시**: 스크린샷, 코드 예제, 데모
5. **Contributing 가이드**: 기여 방법 안내
6. **라이선스**: 오픈소스라면 필수

### 시각적 요소
- 프로젝트 로고
- 상태 배지 (CI, 테스트, 버전)
- GIF 데모

### 주의사항
- API 키를 코드에 직접 저장 금지
- .env 파일 사용, .gitignore에 추가

## 3. 스프린트 회고 (Retrospective)

### 개요
- 스프린트 종료 후, 다음 스프린트 계획 전 수행
- 무엇이 잘 됐고, 안 됐고, 개선할 수 있는지 논의
- 지속적 개선의 핵심 활동

### 회고 템플릿 유형

#### Start, Stop, Continue
- **Start**: 새로 시작할 것
- **Stop**: 멈출 것
- **Continue**: 계속할 것

#### Glad, Sad, Mad
- **Glad**: 기뻤던 것
- **Sad**: 슬펐던 것
- **Mad**: 화났던 것

#### 4Ls
- **Liked**: 좋았던 것
- **Learned**: 배운 것
- **Lacked**: 부족했던 것
- **Longed for**: 원했던 것

### 회고 효과
- 팀 사기 파악
- 프로세스 개선점 발견
- 다음 스프린트 계획에 반영

## 4. 코드 리뷰 체크리스트

### 기능 검증
- [ ] 요구사항대로 동작하는가?
- [ ] 엣지 케이스가 처리되었는가?
- [ ] 에러 처리가 적절한가?

### 코드 품질
- [ ] 코드 컨벤션을 따르는가?
- [ ] 불필요한 코드가 없는가?
- [ ] 변수/함수명이 명확한가?

### 보안
- [ ] 하드코딩된 시크릿이 없는가?
- [ ] 입력 검증이 되어 있는가?
- [ ] XSS/SQL 인젝션 취약점이 없는가?

### 성능
- [ ] 불필요한 렌더링이 없는가?
- [ ] N+1 쿼리 문제가 없는가?
- [ ] 메모리 누수 가능성이 없는가?

## 5. 테스트 전략

### 테스트 피라미드
```
      /\
     /  \  E2E 테스트 (적음)
    /----\
   /      \  통합 테스트 (중간)
  /--------\
 /          \ 유닛 테스트 (많음)
/____________\
```

### 수동 테스트 시나리오
1. 정상 흐름 (Happy Path)
2. 비정상 흐름 (Edge Cases)
3. 에러 상황 (Error Scenarios)
4. 권한 검증 (Authorization)
